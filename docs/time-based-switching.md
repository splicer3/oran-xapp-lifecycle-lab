# Time-Based Switching

Time-Based Switching is the A/B testing strategy used by this lab for `kpimon-go` traffic. It avoids simultaneous percentage splitting and instead alternates all observed RMR/TCP traffic between two xApp versions in fixed time windows.

The runnable workflow lives in `ansible/istio-ab-testing`.

## Why Percentage-Based A/B Testing Is Problematic

Classic A/B testing usually assumes that traffic can be split concurrently, for example 90/10 or 50/50, and that each request can be assigned independently to one variant. That model does not fit this Near-RT RIC setup cleanly.

The xApp traffic path is not a set of short stateless HTTP requests. `kpimon-go` receives RIC messages through RMR over TCP, and the relevant service traffic is configured on TCP ports `4560` and `4561`. A TCP connection can remain attached to one destination for longer than a single message or metric sample. In that case, changing a percentage weight does not mean that individual RIC indications are redistributed immediately or proportionally across xApp versions.

There is also an experiment-design issue. If two xApps act concurrently on the same RIC/RAN context, their effects may not be independent. That violates the SUTVA assumption used by many A/B interpretations: one unit's treatment should not alter another unit's outcome. In a radio-control setting, simultaneous xApps can create spillover through shared subscriptions, shared RAN state, shared RIC components, or common traffic conditions.

Time-Based Switching reduces both issues by using mutually exclusive time windows: one version receives the traffic during a window, then the other version receives it during the next window.

## Strategy

The strategy used in this repository is:

1. Deploy two `kpimon-go` variants, labelled with versions from `versions` in `ansible/istio-ab-testing/group_vars/all.yml`.
2. Build an Istio `DestinationRule` with subsets for those versions.
3. Apply an Istio `VirtualService` for the RMR service host.
4. Route traffic with weights `100/0`.
5. Record an apply marker with timestamp, source version, and target version.
6. Wait for `traffic_cycle_seconds`.
7. Route traffic with weights `0/100`.
8. Record another apply marker.
9. Repeat for `cycles`.
10. Query Prometheus for the whole run and plot the version time series.

By default, the starting version can be randomized with `ab_randomize_start_version: true`. The alternation itself is still deterministic after the start order is selected. Set `ab_randomize_start_version: false` if a fixed `versions[0]` to `versions[1]` order is needed.

## Warm-Up And Measurement Windows

The playbook does not expose separate `warm_up_seconds` or `measurement_window_seconds` variables. It applies the routing change, records the flip marker, waits for `traffic_cycle_seconds`, and then applies the next routing change.

The warm-up and measurement semantics are handled in analysis:

- **Warm-up window**: the early part after a flip where Envoy configuration propagation, TCP/RMR connection drain, Prometheus scrape cadence, and workload readiness can still affect the observed series.
- **Measurement window**: the later part of the same traffic leg, after the old version has drained and the target version is receiving the traffic.
- **Metric markers**: the `flip_events_*.json` records generated by `ansible/istio-ab-testing/playbooks/run_demo.yml`; `scripts/plot_ab.py` uses them as vertical plot markers.

The curated analysis notebook, `experiments/time-based-switching/ab_statistical_validation.ipynb`, defines analysis thresholds and windows for thesis-era samples. Treat those values as analysis policy, not as routing behavior enforced by the playbook.

## Automation Files

The workflow is composed of these files:

| File | Role |
| --- | --- |
| `ansible/istio-ab-testing/playbooks/run_demo.yml` | Orchestrates workload preparation, traffic switching, Prometheus collection, fetch, flip-event persistence, and plotting. |
| `ansible/istio-ab-testing/playbooks/prepare_kpimon_ab.yml` | Verifies the namespace, service, and required ConfigMaps; scales base `kpimon-go` deployments; creates the two A/B deployments. |
| `ansible/istio-ab-testing/playbooks/istio_traffic_ab.yml` | Chooses version order, applies the `DestinationRule`, and runs the flip loop. |
| `ansible/istio-ab-testing/roles/istio_traffic/tasks/flip_pair.yml` | Applies `100/0`, records a marker, waits, applies `0/100`, records a marker, waits. |
| `ansible/istio-ab-testing/roles/istio_traffic/templates/destinationrule.yaml.j2` | Defines Istio subsets by `version` label. |
| `ansible/istio-ab-testing/roles/istio_traffic/templates/virtualservice.yaml.j2` | Generates TCP routing rules for each configured port. |
| `ansible/istio-ab-testing/playbooks/collect_metrics.yml` | Runs the Prometheus `query_range` for `istio_tcp_received_bytes_total`. |
| `ansible/istio-ab-testing/scripts/plot_ab.py` | Converts Prometheus JSON or CSV input into pivoted CSV and Plotly HTML output. |

Important defaults in `ansible/istio-ab-testing/group_vars/all.yml`:

- namespace: `ricxapp`
- service host: `service-ricxapp-kpimon-go-rmr.ricxapp.svc.cluster.local`
- TCP ports: `4560`, `4561`
- versions: `v1`, `v2`
- deployments: `ricxapp-kpimon-go-v1`, `ricxapp-kpimon-go-v2`
- metric: `istio_tcp_received_bytes_total`
- reporter: `destination`
- Prometheus URL: `http://localhost:9090`
- query step: `15s`

## Running The Workflow

Deploy the main RIC lifecycle stack first, then make Prometheus reachable from the Ansible controller at the configured `prometheus_url`.

Use the committed inventory example as a template:

```bash
cd ansible/istio-ab-testing
cp inventory/hosts.ini.example /tmp/istio-ab-hosts.ini
${EDITOR:-vi} /tmp/istio-ab-hosts.ini
ansible-playbook playbooks/run_demo.yml -i /tmp/istio-ab-hosts.ini
```

The run creates ignored local artifacts under `ansible/istio-ab-testing/artifacts/metrics/`, including:

- Prometheus JSON snapshots named like `istio_tcp_*.json`
- flip metadata named like `flip_events_*.json`
- pivot CSV files
- Plotly HTML charts

The sample conclusion reports that, for the analyzed thesis-era runs, the configured transition-latency claim was not statistically supported, while service continuity was statistically supported under the configured dip threshold. This matches the project limitation stated in the README: throughput continuity was preserved in the thesis validation, but convergence latency was the main bottleneck.

## Interpreting Results

Use the generated series to answer narrow questions:

- Did the configured target version eventually receive the RMR/TCP traffic?
- How long did the old version continue receiving traffic after the apply marker?
- Did aggregate throughput dip during the transition?
- Was the post-transition measurement window long enough to compare variants?

Do not interpret a clean `100/0` or `0/100` configuration as instant migration. Istio control-plane propagation, Envoy updates, long-lived TCP/RMR connections, Prometheus scrape interval, and the `rate()` window all affect the observed transition.

Also avoid treating sample CSVs as standalone proof of xApp superiority. In this public repo, both default A/B images are configured as `splicer3/kpi-mon-xapp:1.0.1`; the switching workflow validates routing and measurement mechanics unless the operator intentionally supplies different image tags.

## Known Limitations

- The switching strategy trades simultaneous percentage splitting for time separation.
- The current automation alternates full routing weights but does not enforce separate warm-up and measurement windows at runtime.
- Convergence latency is the main bottleneck observed in the thesis validation.
- RMR/TCP persistence can delay the visible drain of the old destination.
- Prometheus scrape cadence and query range windows can make transition timing appear coarser than the underlying routing event.
- The workflow measures sidecar-observed TCP/RMR traffic. It does not observe SCTP/E2 payloads directly.
- Results depend on the selected xApp images, RIC release, cluster load, sidecar readiness, and Prometheus configuration.

## Future Work

Useful improvements would be:

- Add explicit warm-up and measurement-window variables to the run metadata.
- Emit structured per-leg metadata that records intended active version, apply time, warm-up end, measurement start, and measurement end.
- Calibrate Prometheus scrape interval and query windows for the expected switch cadence.
- Investigate controlled TCP/RMR connection drain or reset behavior to reduce convergence latency.
- Add cleanup playbooks for restoring the base `kpimon-go` deployment state after A/B runs.
- Extend the analysis notebook to consume public sample fixtures directly and document all required artifact schemas.
- Test with genuinely different xApp images once their behavior and provenance are documented.
